# CodexLoop

`codexloop` is a small, Codex-native alternative to Mycelium.

It keeps the useful parts:

- turn an implementation plan into explicit tasks with `codex exec --output-schema`
- run each task in its own Git worktree and branch
- preserve the Codex session ID so interrupted tasks can be resumed
- gate merges with task verification commands plus a repo-local doctor hook
- keep `docs/plans/` as the visible plan and progress surface
- feed `MEMORY.md` back into later Codex turns so solved failures are not repeated
- keep runtime state under `.codexloop/` instead of spreading orchestration state across the repo

It deliberately skips Mycelium's heavier layers such as Docker workers, UI, control-graph tooling, and validator fleets.

## Quickstart

```bash
~/.codex/bin/codexloop init /path/to/target-repo
```

Edit these files in the target repo:

- `.codexloop/config.json`
- `.codexloop/doctor.sh`
- `docs/plans/implementation-plan.md`
- `docs/plans/CODEXLOOP_AGENT.md`
- `MEMORY.md`

These are generated by `codexloop init`. You do not create them manually first.

Generate tasks:

```bash
~/.codex/bin/codexloop plan --repo /path/to/target-repo
```

Run the backlog:

```bash
~/.codex/bin/codexloop run --repo /path/to/target-repo
```

If a task is interrupted or fails verification, resume it:

```bash
~/.codex/bin/codexloop resume --repo /path/to/target-repo
```

Inspect run state:

```bash
~/.codex/bin/codexloop status --repo /path/to/target-repo
```

## How it works

`plan` asks Codex to turn an implementation plan into a compact task graph. `run` creates an integration branch plus one task branch/worktree per task. Codex edits inside the task worktree only, sees the current active plan from `docs/plans/active/`, and gets `MEMORY.md` fed back into the prompt on later turns. After verification passes, `codexloop` creates a commit if needed and fast-forwards the integration branch to include the task branch.

The default execution model is intentionally sequential. Verification failures trigger another Codex turn automatically up to a per-task attempt limit, so the harness keeps pushing until the plan is done or the task really needs human intervention. If you need parallel work later, that should be added on top of the current state model rather than mixed into the first implementation.
